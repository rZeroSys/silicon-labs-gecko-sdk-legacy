3 ## Silicon Labs Connect Stack
4 
5 Silicon Labs is developing products designed to meet the demands of customers moving to a connected world of devices in
6 the home, often referred to as the IoT (Internet of Things). At a high level, the IoT goals for Silicon Labs are as follows:
7 
8 * Connect all devices in the home with best-in-class mesh networking, either with Ember ZigBee PRO or other emerging standards.
9 * Leverage the company expertise in low-power, constrained devices.
10 * Enhance established low-power, mixed-signal chips.
11 * Provide low-cost bridging to existing Ethernet and Wi-Fi devices.
12 * Enable cloud services and connectivity to smartphones and tablets that promote ease of use and a common user experience for
13 customers.
14 
15 Achieving these goals increases adoption rates and user acceptance for IoT devices in the Connected Home, but also involves certain challenges.
16 
17 One challenge is managing devices requiring low-power consumption, such as battery-powered devices where long battery life is
18 essential. To meet this challenge, Silicon Labs has developed the Silicon Labs Connect stack. Connect SDK provides a fully-featured, easily-customizable
19 wireless networking solution optimized for devices that require low power consumption and are used in a simple network
20 topology.
21 
22 Another challenge is to extend an existing, IEEE 802.15.4 based proprietary network. Connect stack's MAC mode provides full access to the MAC layer, which makes it fully compatible with other IEEE 802.15.4-compliant networks.
23 
24 Connect stack includes the following functionalities:
25 
26 * It can work with a wide range of radio configurations and is configurable to be compliant with regional communication standards worldwide. Simplicity Studio is shipped with canned configurations designed for maximum performance under each regional standard.
27 * The stack can operate simple network topologies, which can include sleepy devices or applications can leverage Connect lower layers and build custom routing systems on it.
28 * It also includes a built-in support for Application Framework, which provides a basic toolset for embedded development, such as scheduling or memory allocation.
29 * The Connect stack is part of the Silicon Labs Flex SDK (Software Development Kit), installed through Simplicity Studio. Connect runs
30 on top of RAIL (Radio Abstraction Interface Layer), also included with the Flex SDK. RAIL provides an intuitive, easily-customizable
31 radio interface layer that is designed to support proprietary or standards-based wireless protocols. For more information, see [UG103.13: RAIL Fundamentals](https://www.silabs.com/documents/public/user-guides/ug103-13-app-dev-fundamentals-rail.pdf).
32 
33 With all this functionality already implemented in the stack, users can focus on their end application development and not worry
34 about the lower-level radio and network details.

36 ### Modes of Operation
37 
38 When a Connect device joins a network, it must choose its mode of operation. Inter-operation between these modes is not supported. Therefore, users should choose the mode early in the design stage. The following modes are supported:
39 
40 #### Extended Star Mode
41 
42 * A PAN coordinator (**blue** node in the figure below) administers a centralized network, permitting joining by end nodes (**white** nodes - can be sleepy) and range extenders (**red** nodes) to realize Point-to-Point, Star, and Extended Star networks
43 * The Network layer supports message routing between any two devices (and multiple endpoints at those devices) in the network
44 * Multiple built-in features ("plugins") provided by the stack and/or application framework.
45 
46 #### Direct Mode
47 
48 * A mode for proprietary applications that prefer more complete control of network behavior
49 * Supports connections between devices that are in range of each other (**green** nodes below) - any such device with correct PAN parameters can join
50 * The Network layer supports endpoints, but provides no routing "out of the box" - optimized routing can be implemented in the application
51 * Multiple built-in features ("plugins") provided by the stack and/or application framework
52 
53 #### MAC Mode
54 
55 * A mode (essentially: the Connect MAC layer) for applications that need to be fully IEEE 802.15.4 compatible and inter-operable (when used with a 15.4-compliant radio configuration)
56 * Supports connections between devices (can be sleepy) that are in range of each other (**green** nodes below)
57 * Connect Network layer is not implemented (therefore, no endpoints and fewer built-in features available)
58 * Uses a more complex API than other modes
59 * Some Application Framework plugins (e.g., OTA, Mailbox) won't work out of the box

72 ### Stack Structure
73 
74 The Connect stack provides code organized into three functional layers, as shown in the following figure:
75 
76 ![Stack layers](stack_struct.PNG)
77 
78 The PHY layer must use the frame format defined by IEEE 802.15.4 (1 Byte PHR), but otherwise fully configurable with the radio configurator. For more details, see [AN971](https://www.silabs.com/documents/public/application-notes/an971-efr32-radio-configurator-guide.pdf).
79 
80 The MAC layer is based on the [IEEE 802.15.4-2011 standard](https://standards.ieee.org/standard/802_15_4-2011.html). In MAC mode, the layer is fully compliant with the standard (note that the PHY layer must be configured to be compliant as well). In Extended Star and Direct mode, the MAC layer is not fully compliant with the standard.
81 
82 The Network layer is based on a proprietary protocol.
83 
84 For more details about the stack layers and features, see [UG235.03](https://www.silabs.com/documents/public/user-guides/ug235-03-architecture-of-connect.pdf).
85 
86 ### Application Layer
87 
88 Finally, the Connect Application Framework provides a complete tool and API infrastructure over the underlying stack layers. The Application Framework provides a basic toolset to develop embedded applications, including a [simple task scheduler](@ref event), a [dynamic memory manager](@ref memory_buffer), and a [non-volatile memory manager](@ref tokens). Some functionality within the Application Framework and the Connect stack layers is provided in the form of individual building blocks called plugins. For more details about the plugins for each layer, see the Silicon Labs Connect Application Framework API Reference included in the stack documentation.
89 
90 For more information about Connect Stack, see [UG103.12](http://www.silabs.com/support%20documents/technicaldocs/ug103-12-fundamentals-connect.pdf).
